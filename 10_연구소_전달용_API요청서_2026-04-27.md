# 연구소 협의 요청 — 브랜치Q 고객만족도 평가 기능

> **작성일**: 2026-04-27
> **요청자**: 김병율 부장 (브랜치Q 개발)
> **대상**: 연구소 8200 / chatadmin 담당자
> **요약**: 브랜치Q 응답화면에 고객만족도 평가 기능을 추가 개발 중입니다. 신고하기(`/api/report`) 와 동일 패턴으로 신규 엔드포인트 1건 + 어드민 페이지 1건의 신설을 요청드립니다.

---

## 배경

브랜치Q 챗봇 답변에 대한 사용자 만족도(별점 1~5 + 사유 다중 선택 + 자유 의견) 를 수집하여, 부정 평가의 근본 원인 분석과 답변 품질 개선의 첫 단계를 만들고자 합니다.

기존 신고하기 기능과 데이터 흐름이 거의 동일하므로 신고하기 인프라 그대로 복제하시면 됩니다.

---

## 1. 백엔드 API 신설 — `POST /api/feedback`

기존 `POST /api/report` (신고하기) 와 동일한 인증·응답 패턴의 신규 엔드포인트 추가 요청드립니다.

| 항목 | 값 |
|---|---|
| 메소드 | `POST` |
| 경로 | `/api/feedback` |
| Content-Type | `multipart/form-data` — 신고하기(`/api/report`)와 동일 (향후 만족도에 첨부 추가 시에도 규약 변경 불요) |
| 인증 | 신고하기와 동일 — 요청 헤더 `JWT-Token` 검증 |
| 처리 | DB 저장 후 신고하기와 동일 형태의 응답 |

### 페이로드 (FormData 필드 — 신고하기 식별 필드 그대로 + 만족도 신규 필드)

```
[ 신고하기 saveVocApi 와 동일한 식별 필드 ]
userId:            ai_hanaro@webcash.co.kr
companyId:         1078783749
companyNm:         (주)웹케시
prodCd:            kjb
prodNm:            AI-광주은행
useInttId:         1078783749
channel:           PC
targetCorpNo:      1078783749
workflowId:        abc-123-def
utteranceContents: [사용자 질문] ...
                   [에이전트 답변] ...

[ 만족도 신규 필드 ]
score:             4
reasons:           ["accurate","helpful"]
comment:           한도 잔여 비율 표현이 직관적이지 않아요
```

> 전송 방식은 `multipart/form-data` (신고하기와 동일 인스턴스 `formDataApi`). 향후 첨부 추가 시 신고하기처럼 `uploadfiles` 필드를 추가하면 됩니다.
>
> `reasons` 의 직렬화 형태(JSON 문자열 / 콤마 구분 / 다중 append 등)는 연구소 측 처리 표준에 맞춰주시면 됩니다. 위 예시는 JSON 문자열 형태입니다.

### 필드 정의

**기존 신고하기와 동일한 식별 필드 — 동일 의미·동일 출처(localStorage)**

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| userId | string | O | 사용자 ID |
| companyId | string | O | 회사 ID |
| companyNm | string | O | 회사명 |
| prodCd / prodNm | string | O | 상품 코드 / 이름 |
| useInttId | string | O | 회사 인증 ID |
| channel | string | O | `"PC"` 고정 |
| targetCorpNo | string | O | 대상 사업장 번호 |
| workflowId | string | O | **답변 메시지 ID** — 평가 대상 식별자 (어드민 디테일 모달 매핑용) |
| utteranceContents | string | △ | 사용자 질문 + AI 답변 컨텍스트 (어드민 디테일에서 답변 전문 표시용) |

**만족도 전용 신규 필드**

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `score` | int | O | 별점 1~5 |
| `reasons` | string[] | O | 다중 선택 사유 (빈 배열 허용). 클라이언트가 점수에 따라 긍정/부정 코드값 풀로 분기 송신 (아래 표 참조) |
| `comment` | string \| null | △ | 자유 의견 (200자 이내, 선택사항). 미입력 시 `null` 또는 미포함 |

**reasons 코드값 풀** (클라이언트 점수별 분기 — 백엔드는 배열 그대로 저장)

| 점수 | 코드값 | 라벨 |
|---|---|---|
| 4~5점 (긍정) | `accurate` | 정확해요 |
| | `helpful` | 도움이 됐어요 |
| | `fast` | 빨랐어요 |
| | `easy` | 이해하기 쉬워요 |
| | `other` | 기타 |
| 1~3점 (부정) | `inaccurate` | 부정확해요 |
| | `insufficient` | 내용이 부족해요 |
| | `slow` | 느렸어요 |
| | `hard` | 이해하기 어려워요 |
| | `other` | 기타 |

> 코드값 네이밍은 클라이언트 측 제안값입니다. 연구소 측 표준에 맞춰 조정 가능합니다.

> 신고하기와의 차이: 신고하기 페이로드 중 `question` / `content` / `uploadfile` 은 만족도에 사용하지 않습니다. 평가 정보는 `score` / `reasons` / `comment` 로 구조화됩니다. 분류 식별은 라우트(`/api/feedback`) 자체로 충분하므로 별도 type 필드는 두지 않았습니다 — 필요 시 연구소 측 표준에 맞춰 추가 가능합니다.

### 응답 스키마 (신고하기와 동일)

```json
{
  "httpStatus": "200",
  "status":     "success",
  "success":    true,
  "message":    "정상 처리되었습니다",
  "data":       {}
}
```

> 신고하기 응답 타입(`AdminDaquvResponse`) 그대로. `data` 안에 부가 정보(예: 저장 ID) 포함 여부는 연구소 측 판단에 맡기겠습니다.

## 2. DB 테이블 — `feedback` (제안)

신규 테이블 신설 (또는 기존 신고하기 테이블에 컬럼 추가 — 연구소 측 판단). 아래는 PostgreSQL 문법 기준 제안이며, 테이블명·컬럼명·DBMS는 연구소 표준에 맞춰 조정해 주시면 됩니다:

```sql
-- PostgreSQL 문법 기준 예시. DBMS / 명명 규칙 / 제약조건은 연구소 표준으로 조정 부탁드립니다.
CREATE TABLE feedback (
  feedback_id     BIGSERIAL PRIMARY KEY,
  user_id         VARCHAR(255) NOT NULL,
  company_id      VARCHAR(50)  NOT NULL,
  company_nm      VARCHAR(255),
  prod_cd         VARCHAR(50),
  prod_nm         VARCHAR(100),
  use_intt_id     VARCHAR(50),
  channel         VARCHAR(10),
  target_corp_no  VARCHAR(50),
  workflow_id     VARCHAR(100) NOT NULL,
  score           SMALLINT     NOT NULL,                  -- 1~5
  reasons         VARCHAR(255),                            -- 다중 사유 직렬화
  comment         TEXT,                                    -- 자유 의견 (선택)
  utterance       TEXT,                                    -- 사용자 질문 + AI 답변 컨텍스트
  status          VARCHAR(20),                             -- 처리상태 (어드민 운영 X — A3 결정으로 미사용. 컬럼 유지/제거는 연구소 판단)
  created_at      TIMESTAMP    DEFAULT now(),
  processed_at    TIMESTAMP                                 -- 어드민 처리상태 미운영으로 사용 안 함 (컬럼 유지/제거는 연구소 판단)
);
```

> 위는 어디까지나 페이로드 → 컬럼 매핑 예시입니다. 다음은 연구소 측 정책 결정사항으로 위임드립니다:
> - 인덱스 (회사·workflow 등 조회 패턴)
> - 중복 제출 방지 정책 (UNIQUE 제약 또는 애플리케이션 레벨 dedup)
> - reasons 직렬화 형식 (JSON 문자열 / 콤마 / 별도 1:N 테이블)
> - 처리상태 기본값 / 코드 체계
> - 신고하기 테이블에 컬럼 추가하여 통합할지, 별도 테이블로 분리할지

## 3. 어드민 페이지 신설 — `/admin/feedback`

`https://cloudbranchq-chatadmin-dev.aiwebcash.co.kr/admin/feedback` (경로명은 연구소 표준에 맞춰 조정 가능).

기존 VOC 어드민(`/admin/voc`) 의 패턴을 참고하여 만족도 데이터 조회 화면을 신설 부탁드립니다.

### 목록 화면 — 컬럼 권장사항 (참고)

VOC 어드민에서 친숙하실 표시 항목 외에 만족도용으로 추가되는 항목:

| 항목 | 출처 필드 | 표시 권장 |
|---|---|---|
| 점수 | `score` | ★ 1~5 시각 표시 |
| 사유 | `reasons` | 라벨 join 예: `정확해요 · 도움이 됐어요` |
| 자유 의견 | `comment` | 1줄 미리보기 (있는 경우만) |
| 등록일시 / 회사명 / 사용자 | 기존 패턴 | VOC와 동일 |

> ⚠️ **처리완료 토글/처리상태 컬럼 제외** (기획자 결정 A3, 2026-04-27) — 만족도 데이터는 처리상태 운영이 불필요하다는 결정. 기존 신고하기 어드민과 달리 처리완료 토글 / 처리상태 컬럼 / 처리상태 필터를 두지 않습니다.

### 디테일 (참고)

VOC 디테일과 유사하게 **답변 전문 + 평가 정보** 구성을 권장드립니다 (처리 토글 없이). 구성 항목:
- 사용자 질문 + AI 답변 전문
- 점수 (★ 1~5)
- 사유 (`reasons` 라벨 join)
- 자유 의견 (`comment`)

답변 전문 매핑은 `workflowId` 또는 페이로드의 `utteranceContents` 활용 가능 — 어느 방식이 연구소 측 데이터 모델과 맞는지에 따라 선택 부탁드립니다.

### 필터 (참고)

VOC 어드민에 있는 필터(날짜/회사 등)에 더해 다음을 권장드립니다 — 추가 여부는 연구소 판단에 맡기겠습니다:
- 점수 범위 필터 (부정 평가만 추리는 용도)

## 4. 기존 인증·보안·보관 정책 (변경 없음)

| 항목 | 처리 |
|---|---|
| `/api/feedback` 인증 | 기존 `/api/report` 와 동일 — `JWT-Token` 헤더 필수 |
| 인증 예외 화이트리스트 | 추가 등록 불요. 모든 호출은 로그인 사용자 (공유 기능 같은 비로그인 예외 경로 아님) |
| 사용자 식별 | 페이로드에 `userId` / `companyId` 포함 (기존 신고하기 패턴 그대로) |
| 어드민 권한 | 기존 VOC 어드민과 동일 권한 그룹 적용 |
| **데이터 보관 정책** | **신고하기와 완전히 동일** — 식별 필드 + 답변 전문(`utteranceContents`) + 평가 3종(`score`/`reasons`/`comment`) 전체 저장. 별도 마스킹·만료 룰 신설 불요. (기획자 결정 A6, 2026-04-27) |
| **위젯 노출 룰** | **신고하기 버튼 노출 룰과 동일** — supervisor 필터링된 응답에서는 신고하기·만족도 모두 미노출. 별도 supervisor 식별 플래그 신규 불요. (기획자 결정 A7, 2026-04-27) |

## 5. 우리(브랜치Q 프론트) 측 작업 (참고용)

연구소 측 작업이 아닙니다. 참고만 부탁드립니다.

| 항목 | 내용 |
|---|---|
| 진입 위치 | 보고서 응답의 [신고하기] 버튼 옆 + FAQ 응답의 [신고하기] 버튼 옆 (모두 dev 환경 한정) |
| 모달 컴포넌트 | `apps/client/src/components/feedback/FeedbackModal/FeedbackModal.tsx` (별점 + 사유 칩 + 자유 의견 + 제출) |
| 모달 호출 | `apps/client/src/components/answers/aiResponse/AIResponse.tsx` (보고서·FAQ 두 경로에서 공유) |
| 환경 가드 | `getEnvMode() === 'development'` (운영 도메인 자동 차단) |

호출은 기존 `formDataApi` 인스턴스 사용 — `formDataApi.post('/feedback', formData)` (`baseURL = import.meta.env.VITE_API_URL = /api`, 신고하기 `saveVocApi` 와 동일 인스턴스).

현재 우리 측은 mock(`console.log`) 으로 동작 중입니다. 연구소 `/api/feedback` 가동 후 신고하기 `saveVocApi` 패턴으로 신규 함수(`saveFeedbackApi`) 작성 + FeedbackModal 의 mock 호출 부분을 교체할 예정입니다.
