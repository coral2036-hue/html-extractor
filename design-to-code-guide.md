## 1. 디자인 코딩의 표준: TDS(Toss Design System) 이해

TDS는 단순한 가이드라인이 아니라, **디자인과 개발이 '하나의 언어'를 사용하도록 설계된 소프트웨어 제품**입니다.

### TDS의 시작과 역사

| 시기 | 내용 |
|------|------|
| **2018년 하반기** (탄생) | 토스가 급격히 성장하며 서비스가 파편화되자, '속도'와 '일관성'을 동시에 잡기 위해 엔지니어링 관점에서 디자인을 재정의하기 시작 |
| **2019년 초** (전면 도입) | 단순 문서화를 넘어, 코드로 구현된 컴포넌트 라이브러리가 전 사 서비스에 적용 |
| **현재** (진화) | 모바일 앱뿐만 아니라 웹, 오프라인(토스플레이스)까지 아우르는 거대한 생태계로 발전. AI가 이 시스템의 규칙을 학습하여 코드를 생성하는 단계까지 도달 |

### TDS의 3층 구조 (Structure)

```
┌─────────────────────────────────────┐
│  Patterns (레이아웃)                 │  ← 컴포넌트들이 배치되는 표준화된 방식
├─────────────────────────────────────┤
│  Components (부품)                   │  ← 토큰이 조합된 버튼, 입력창 등
├─────────────────────────────────────┤
│  Foundations (디자인 토큰)            │  ← 색상, 간격, 타이포그래피 등 최소 단위 수치
└─────────────────────────────────────┘
```

- **Foundations (디자인 토큰)**: JSON 형태로 관리되어 Figma와 코드(Tailwind 등)가 동일한 변수명을 공유
- **Components (부품)**: 개발자는 스타일을 짜는 게 아니라 **컴포넌트를 호출**
- **Patterns (레이아웃)**: 컴포넌트들이 배치되는 표준화된 방식

## 2. Dev: 코드로 구현되는 시스템 (Developer Experience)

개발자에게 디자인 시스템은 **고민을 줄여주는 강력한 도구**입니다.

- **NPM 라이브러리화**: TDS는 내부 패키지로 배포되어 `npm install` 한 번으로 검증된 UI를 즉시 사용
- **Storybook 활용**: 실제 구현 전 컴포넌트의 다양한 상태(Loading, Error 등)를 미리 테스트하고 API 명세를 확인
- **Semantic API**: `size="large"`, `variant="primary"` 등 기능적 의미를 담은 속성을 사용

### 🚀 AI 시대의 디자인 시스템: TDS라는 훌륭한 프로토타입

AI 에이전트가 개발의 주역이 되는 시대에, 잘 정립된 TDS는 가장 완벽한 **프로토타입(Prototype)** 역할을 합니다.

> **벤치마크를 통한 고도화**: TDS 구조를 벤치마킹하여 프로젝트만의 고유한 문법으로 재해석하면 훨씬 고도화된 시스템 구축이 가능합니다.

> **AI와의 시너지**: AI는 명확한 규칙 위에서 가장 강력합니다. TDS의 논리적 구조는 AI가 의도를 100% 이해하게 만드는 '미래 지향적 설계'의 시작점입니다.

> **효율적인 이정표**: 시스템 도입을 강제하는 것이 아니라, 디자인을 코드로 치환하는 과정에서 시행착오를 줄여주는 효율적인 이정표입니다.

## 3. 디자인 품질/중요도별 워크플로우 비교표

> 모두가 Figma를 쓸 수 없기에, **바이브 코딩(Vibe Coding)** 을 통한 1차 디자인 방식은 매우 효율적인 대안입니다.

| 구분 | (1) 품질 최우선 | (2) 품질 중상 | (3) 품질 중 | (4) 간단 디자인 |
|------|--------------|------------|-----------|--------------|
| **대상** | B2C / 브랜딩 | 서비스 런칭 | SI / 기업용 | PoC / 어드민 |
| **핵심 목표** | 완벽한 픽셀과 독창적 UX | 빠른 시장 검증 및 심미성 | 기능 중심의 일관성 보장 | 속도 및 데이터 전달력 |
| **워크플로우** | Pencil → Figma → Dev | Pencil → Vibe Coding → Dev | AI → Tailwind System | AI → 즉시 배포 |
| **주요 도구** | 전담 디자이너, 정교한 TDS | v0.dev, Claude, shadcn/ui | v0.dev, Tailwind UI | Claude Artifacts, Bolt |
| **AI의 역할** | 코드 구현 보조 (Copilot) | 디자인 초안 및 UI 코드 생성 | 레이아웃 전체 초안 생성 | 기획부터 코드까지 전체 수행 |
| **Figma 위상** | 필수 (설계 거점) | 선택 (최종 검수/교정용) | 거의 없음 | 없음 |

### 주요 도구 상세 설명 (Tool-Chain)

| 도구 | 설명 |
|------|------|
| **Pencil** (UX Blueprint) | 아이디어를 빠르게 가시화하고 AI Vision을 통해 설계도로 변환하는 시작점 |
| **Vibe Coding** (AI-Driven Design) | Figma 대신 AI와 대화하며 실시간으로 동작하는 HTML/React 목업을 만들어내는 단계 |
| **v0.dev / Claude Artifacts** | 프롬프트를 기반으로 즉시 실행 가능한 React/Tailwind 코드를 생성하는 UI 특화 에이전트 |
| **shadcn/ui** | 디자인 시스템을 코드로 구현할 때 가장 유연하게 대응할 수 있는 표준 컴포넌트 킷 |
| **Claude Code / MCP** | 전체 컨텍스트를 이해하고 디자인 토큰과 스케치를 결합하여 코드를 직접 작성하는 오케스트레이터 |

## 4. 고효율 AI-Native 워크플로우 (사용자 고안 방식)

기획부터 개발까지 AI 에이전트를 최대한 활용하여 디자인을 코드로 치환하는 **최적의 5단계 프로세스**입니다.

```
① 아이디어 가시화  →  ② 설계 구조화  →  ③ 마스터 컨텍스트 구축  →  ④ AI 주도 개발  →  ⑤ 정밀 검수
```

---

### Step 1. 아이디어 가시화 (Pencil & Vibe Coding)

- **도구**: Pencil, Gemini, Stitch
- **결과물**: `.html` 목업 또는 `.pen` 스케치 파일
- **설명**: Figma 대신 바이브 코딩으로 핵심 인터랙션과 배치를 확인합니다. Pencil 스케치를 AI Vision으로 읽어들여 초안 레이아웃을 확정합니다.

---

### Step 2. 설계 구조화 (JSON Sidecar & PNG)

- **도구**: AI 에이전트
- **결과물**: `json_sidecar` (화면 구조 정의) + `.png` (시각적 레퍼런스)
- **설명**: Step 1의 목업을 바탕으로 컴포넌트 트리와 데이터 타입을 정의한 JSON 파일과 시각적 기준이 될 PNG 에셋을 생성합니다.

```json
{
  "screen": "HomeScreen",
  "components": [
    { "type": "Header", "props": { "title": "string", "variant": "primary" } },
    { "type": "CardList", "props": { "items": "Card[]" } }
  ]
}
```

---

### Step 3. 마스터 컨텍스트 구축 (Design Token MD)

- **결과물**: `Design_Token.md` (또는 `.json`)
- **설명**: Step 1~2에서 나온 데이터들을 교차 분석하여 프로젝트 전체에서 쓰일 디자인 토큰을 문서화합니다. 이는 **AI가 코드를 짤 때 참고할 '법전'** 이 됩니다.

> 💡 이 단계가 탄탄할수록 Step 4에서 AI의 코드 품질이 극적으로 향상됩니다.

---

### Step 4. AI 주도 개발 (Implementation)

- **도구**: Claude Code, Cursor
- **설명**: 생성된 Step 1~3 파일들을 AI 에이전트에게 제공합니다. AI는 이 데이터를 바탕으로 시스템 통합 코드를 작성합니다.

```
제공 컨텍스트:
├── step1_mockup.html        # 레이아웃 기준
├── step2_structure.json     # 컴포넌트 트리
├── step2_reference.png      # 시각적 기준
└── step3_design_token.md    # 디자인 법전
```

---

### Step 5. 정밀 구체화 및 검수 (Figma - Optional)

- **도구**: Figma
- **설명**: 모든 화면을 그리는 것이 아니라, AI가 만든 최종 결과물에서 **픽셀 단위의 미세 교정**이나 **브랜드 정체성 확인**이 필요할 때만 선별적으로 활용합니다.
