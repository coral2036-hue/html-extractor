### A3. 만족도 받은 데이터를 어떻게 활용할 것인가

**중요한 전제 (구조 정리)**:
- 사용자(고객사) 화면 = **우리 BranchQ 프론트**에서 만족도를 **등록 받음**
- 받은 데이터를 보고·관리하는 화면 = **연구소가 관리하는 어드민 사이트**에 만들어야 함
  → `https://cloudbranchq-chatadmin-dev.aiwebcash.co.kr/` (예: `/admin/feedback`)
- 즉, **우리 프론트로는 활용 화면을 만들지 않음**. 데이터를 8200(연구소)으로 보내면, 연구소가 자기네 어드민에서 보여줌
- 신고하기(`/admin/voc`)와 **완전히 동일한 룰** — 우리는 등록 API만, 어드민은 연구소가 만듦

**현재 상태**: 어드민 페이지는 **목록 화면만** 기획되어 있음. 한 평가를 클릭했을 때 디테일·부정 평가 처리 등이 정해지지 않음.

**문제**: 사용자에게서 평가를 받아놓고 그 데이터로 무엇을 할지 정해지지 않으면, 기능을 만들어도 활용이 안 됨.

**선택지** (1차 운영 범위 결정):
- **선택지 1**: 목록만 일단 운영 (사용자 평가 데이터만 쌓아두기)
- **선택지 2**: 목록 + **디테일 모달 기본형** (질문/답변 전문 + 점수 + 사유 + 자유 의견 + 처리완료 토글) 까지 함께 (권장)
- **선택지 3**: 위 2 + 통계 대시보드(만족도 추이, 부정 평가 비율) 까지

**결정 필요**: 어디까지 1차 운영범위로 잡을지? 통계 대시보드는 향후로 미뤄도 됩니다.

**필요한 신규 개발 (참고용)**:

| 영역 | 작업 | 담당 |
|---|---|---|
| **연구소 어드민** (`cloudbranchq-chatadmin-dev`) | `/admin/feedback` 페이지 신규 — VOC 어드민(`/admin/voc`) 코드 복제 후 컬럼 교체 | 연구소 |
| **연구소 8200 백엔드** | `POST /api/feedback` 엔드포인트 신설 (`/api/report` 패턴 복제) + DB 테이블 | 연구소 |
| **우리 BranchQ 프론트** | 만족도 위젯 컴포넌트 + `weaiApi.post('/feedback', ...)` 호출 (신고하기 `saveVocApi` 패턴 동일) | hcms |

## 🔴 사용자 경험·정책 결정 — 5건

### A4. 부정 평가 사유를 어떻게 받을 것인가

**현재 상태**: 사유 체크박스(칩)가 5개인데 모두 긍정 표현(정확해요/도움이 됐어요/빨랐어요/이해하기 쉬워요/기타).

**문제**: 사용자가 별 1~2점(불만족)을 줘도 **선택할 사유가 "기타"밖에 없음**. 부정 평가의 원인 분석이 본 기능의 핵심 목적인데, 정작 부정 사유를 칩으로는 수집할 수 없음.

**선택지**:
- **선택지 1**: 점수에 따라 칩이 바뀜
  - 점수 4~5점 → 긍정 사유 칩 (정확해요/도움/빠름/이해/기타)
  - 점수 1~3점 → 부정 사유 칩 (부정확/내용 부족/느림/이해 어려움/기타)
- **선택지 2**: 칩 풀을 중립적으로 (예: "정확성", "속도", "이해도", "내용량", "기타") — 사용자가 어떤 영역에 대한 평가인지만 표시. 만족/불만족은 점수로 표현

**결정 필요**: 1 / 2 중 어느 것?

### A5. 평가 제출 후 수정 가능 여부

**현재 상태**: 한 번 제출하면 잠금 카드가 뜨고, **수정 불가**.

**문제**: 사용자가 별 4점을 클릭하려다 실수로 3점을 누르고 제출했을 때 정정 못함 → 데이터 정확도 ↓.

**전제 (중요)**: 우리 프론트는 **채팅 화면을 벗어나면 이전 대화 내용을 다시 보여주지 않음**. 그래서 "24시간 내 변경" 같은 시간 기반 옵션은 의미가 없음 — 사용자가 그 답변으로 다시 돌아올 수 없기 때문.

**현실적 선택지**:
- **선택지 1**: 지금처럼 즉시 잠금, 수정 불가 (현재 안)
- **선택지 2**: 잠금 카드에 **"수정하기" 버튼**을 두고, 클릭하면 입력 UI로 다시 펼쳐짐 (현재 화면에 머무는 동안만 가능 — 실수 정정 정도가 목적)

**결정 필요**: 1 / 2 중 어느 것?

> ※ 채팅 영속화(나간 후 돌아와도 평가 상태 유지)는 별도 큰 기능 — 본 결정 범위 밖.

### A6. 사용자가 쓴 자유 의견 보관 정책

**현재 상태**: 자유 의견에 사용자가 개인정보(이름, 전화번호 등)를 적을 수 있음. **만족도 데이터 자체의 보관 룰 미정**.

**전제**: 우리 시스템은 이미 **채팅 대화 저장 정책**을 운영 중. 만족도 평가도 채팅의 일부 데이터이므로 동일 룰 적용이 자연스러움 → 별도 정책 신설 불필요.

**확인 필요사항** (선택지 X, 단순 확인):
- **확인 1**: 현재 채팅 저장 정책의 보관 기간이 어떻게 되는지 (예: N년)
- **확인 2**: 채팅에 개인정보 마스킹 처리 룰이 있다면 동일 룰을 만족도 데이터(자유 의견)에도 적용
- **확인 3**: 사용자 삭제 요청 채널이 이미 있다면 같이 적용 (없으면 신규 항목 아님 — 후속)

**결정 필요**: "**기존 채팅 저장 정책 그대로 따름**" 으로 확정하면 끝. 별도 신규 정책 없음.

> ※ 만약 채팅 저장 정책이 명시적으로 없거나 모호하다면, 그때만 별도 정책 마련 필요.

### A7. 모든 답변에 평가 위젯을 보여줄 것인가

**전제**: 만족도 평가는 **일반 질의(FAQ류) 답변에 대한 평가**. "안녕하세요" 같은 인사성 입력은 정상 사용 시나리오가 아닌 **고객 측 잘못된 질의**로 봐야 함 → 그 케이스 때문에 노출 분기를 만들 필요 없음.

**기본 결정 (제안)**: "**모든 정상 답변에 노출**" — 분기 불필요.

**확인 필요사항** (예외 케이스만):
- 시스템이 "비정상/오입력 질의" 를 명확히 식별하는 룰이 있는지? (있으면 그때만 위젯 미노출, 없으면 그냥 모든 답변에 노출)

> ※ 본 항목은 별도 결정 불필요. 단순 확인.

---

### A8. "기타" 칩을 선택했을 때 자유 의견을 필수로 받을 것인가

**현재 상태**: "기타" 선택해도 자유 의견은 선택사항. 결과적으로 **"기타"만 선택하고 끝**도 가능.

**문제**: "기타"가 무엇인지 알 수 없음 → 데이터 가치 ↓.

**선택지**:
- **선택지 1**: "기타" 선택 시 자유 의견 **필수**(빈 칸이면 제출 비활성화) — 권장
- **선택지 2**: 지금처럼 둘 다 선택사항

**결정 필요**: 1 / 2 중 어느 것?

### A9. 점수가 낮은데 사유도 의견도 비어있는 평가는 어떻게 처리할 것인가

**현재 상태**: `점수 1점 + 사유 미선택 + 자유 의견 미입력` 으로도 제출 가능.

**문제**: 어드민 화면에서 "1점인데 왜 그런지 모름" 데이터만 쌓임 → 모델 개선에 활용 불가.

**선택지**:
- **선택지 1**: 점수 3점 이하면 사유 칩 또는 자유 의견 중 1개 이상 **필수** — 권장
- **선택지 2**: 지금처럼 점수만 있어도 제출 가능 (단, 어드민 통계에서 별도 분류)

**결정 필요**: 1 / 2 중 어느 것?

### A10. 자유 의견 입력 글자 수

**현재 상태**: 한 줄 input 박스. **글자 수 제한 미정**.

**문제**: 사용자가 너무 길게 쓰면 어드민 화면이 어지러워지고 저장 비용도 ↑.

**선택지**:
- **선택지 1**: 100자 (한 문장 정도)
- **선택지 2**: 200자 (한두 문장)
- **선택지 3**: 500자 (짧은 단락)

**결정 필요**: 1 / 2 / 3 중 어느 것? (권장: 200자)

## 🅱 우리(프론트) 측 결정 — 6건

### B5. 모바일 반응형 (< 360px)
- 별점 5개 + 칩 5개 자동 wrap (2줄 허용)
- 칩 폰트·padding 자동 축소
- 디자인 시스템 토큰 재사용

### B6. 접근성 (a11y)
- 별점 키보드 ←→ 이동
- screen reader: `"4점 / 5점 만점 / 선택됨"` announce
- focus-visible 스타일

### B7. 에러 처리 (신고하기 패턴 동일 적용)
- `useFeedback` 훅의 `onError` 핸들러에서 토스트 표시
- 신고하기와 동일하게 throw 후 컴포넌트가 받아서 처리
- (재시도/오프라인 큐는 신고하기에도 없음 — 동일 정책 유지)

### B8. emoji → SVG 아이콘 전환
- ⭐ ✓ ✅ 텍스트 emoji 대신 SVG 아이콘 토큰화
- 디자인 시스템에 추가

### B9. 글자 수 측정 단위 (A10 결정 후)
- comment maxLength 적용 시 grapheme cluster 단위 (이모지·한글 자모 분리 대응)

### B10. ESC 키 동작
- 입력 reset 또는 무동작 — 신고하기 모달에 ESC 처리 없으므로 동일하게 무동작
